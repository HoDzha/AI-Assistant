# Этап 3. Интеграция LLM

## 1. Цель этапа

Цель данного этапа заключается в подключении большой языковой модели (LLM) к системе управления задачами и в разработке промптов, которые позволят анализировать список задач, определять их приоритет и формировать план выполнения работы.

На этом этапе ИИ-модель становится центральным аналитическим компонентом системы. Именно она обрабатывает список задач, учитывает дедлайны, важность и дополнительные параметры, а затем формирует структурированный ответ, полезный для фрилансера.

## 2. Роль LLM в проекте

В рамках проекта LLM используется как интеллектуальный модуль, который выполняет следующие функции:

- анализирует список задач;
- определяет приоритетность задач;
- выявляет срочные и важные задачи;
- предлагает последовательность выполнения;
- формирует рекомендации по организации рабочего процесса;
- создает план работы на день или на ближайший период.

Использование LLM позволяет сделать систему более гибкой по сравнению с жестко заданными правилами, так как модель может учитывать контекст задачи и текстовые комментарии пользователя.

## 3. Подключение LLM через API

### 3.1. Общая схема интеграции

Подключение LLM через API предполагает следующий процесс:

1. Пользователь вводит задачи через интерфейс.
2. Данные проходят предварительную валидацию и преобразование в JSON.
3. Python-модуль подготавливает запрос к LLM.
4. В запрос включаются:
   системная инструкция, пользовательские данные и промпт.
5. Запрос отправляется в API модели.
6. Система получает ответ от LLM.
7. Результат преобразуется в удобный вид и выводится пользователю.

Таким образом, LLM выступает внешним сервисом интеллектуальной обработки, а приложение отвечает за подготовку данных, отправку запроса и обработку результата.

### 3.2. Компоненты интеграции

Для подключения LLM через API в проекте необходимы следующие компоненты:

- `Python` как основной язык реализации;
- библиотека для HTTP-запросов или SDK провайдера API;
- API-ключ для доступа к модели;
- модуль формирования промптов;
- модуль обработки ответов модели.

### 3.3. Передаваемые данные

При обращении к LLM система должна передавать:

- список задач;
- дедлайны;
- оценку важности;
- ссылку на GitHub-репозиторий проекта, если она указана;
- длительность выполнения;
- статус задач;
- дополнительные комментарии;
- доступное количество рабочих часов.

Эти данные передаются в структурированном виде, чтобы модель могла корректно интерпретировать входную информацию.

### 3.4. Пример логики обращения к API

На уровне логики приложение должно выполнить следующие действия:

1. Получить данные из интерфейса.
2. Преобразовать задачи в единый формат.
3. Сформировать текстовый промпт.
4. Отправить запрос в API LLM.
5. Получить ответ.
6. Проверить структуру ответа.
7. Вывести пользователю приоритеты и план работы.

### 3.5. Пример структуры API-запроса

Ниже приведен обобщенный пример структуры запроса:

```json
{
  "model": "llm-model-name",
  "messages": [
    {
      "role": "system",
      "content": "Ты ИИ-ассистент по управлению задачами фрилансера. Анализируй задачи, определяй приоритет и формируй план работы."
    },
    {
      "role": "user",
      "content": "Вот список задач: ... Сформируй структурированный анализ."
    }
  ]
}
```

Точная структура зависит от используемого API, однако логика остается одинаковой: система передает инструкцию, контекст и список задач, а модель возвращает результат анализа.

### 3.6. Требования к обработке ответа

Ответ LLM должен быть:

- структурированным;
- понятным для дальнейшей программной обработки;
- пригодным для отображения в интерфейсе;
- включающим список задач со статусами, если это требуется логикой интерфейса;
- устойчивым к повторному использованию в системе.

Желательно, чтобы модель возвращала результат в JSON-формате или в четко заданной структуре с фиксированными разделами.

## 4. Разработка промптов для анализа задач

### 4.1. Назначение промптов

Промпт в данном проекте играет ключевую роль, так как именно он задает модели:

- роль ассистента;
- цель анализа;
- критерии определения приоритетов;
- требования к формату ответа.

Хорошо составленный промпт позволяет получать более стабильные, полезные и предсказуемые результаты.

### 4.2. Основные требования к промптам

Промпты должны:

- задавать модели роль ассистента по управлению задачами;
- учитывать дедлайны, важность и длительность задач;
- просить модель ранжировать задачи;
- требовать объяснения причин приоритизации;
- формировать итоговый план выполнения;
- возвращать результат в структурированном формате.

### 4.3. Базовая структура промпта

Для проекта целесообразно использовать промпт из трех частей:

1. Роль модели.
2. Контекст задачи.
3. Инструкция по формату ответа.

Пример общей структуры:

- роль: ассистент по управлению задачами фрилансера;
- вход: список задач с параметрами;
- задача: проанализировать, определить приоритет и составить план;
- результат: выдать структурированный ответ.

## 5. Промпт для анализа задач

Ниже приведен пример промпта для анализа списка задач:

```text
Ты выступаешь в роли ИИ-ассистента по управлению задачами фрилансера.

Твоя задача:
1. Проанализировать список задач.
2. Определить приоритет каждой задачи.
3. Учесть сроки выполнения, важность, длительность и зависимости.
4. Выявить самые срочные и критичные задачи.

Правила анализа:
- высокий приоритет получают задачи с близким дедлайном и высокой важностью;
- если задача блокирует другие задачи, это нужно учитывать;
- если задача просрочена или почти просрочена, отметь это;
- учитывай ограничение по рабочему времени пользователя.

Верни результат в следующем формате:
- задача;
- статус задачи;
- GitHub-ссылка проекта;
- итоговый приоритет;
- причина назначения приоритета;
- рекомендуемое место в порядке выполнения;
- возможный риск при откладывании.
```

### 5.1. Назначение этого промпта

Данный промпт помогает модели сосредоточиться именно на анализе задач и их приоритизации. Он задает четкие критерии оценки и снижает вероятность слишком общего ответа.

## 6. Промпт для формирования плана выполнения задач

После анализа задач можно использовать отдельный промпт для построения плана работы.

```text
Ты ИИ-ассистент по планированию работы фрилансера.

На основе уже проанализированного списка задач составь план выполнения задач на рабочий день.

Условия:
- учитывай доступное количество рабочих часов;
- начни с самых срочных и важных задач;
- не перегружай день задачами сверх доступного времени;
- если все задачи не помещаются в один день, предложи очередность переноса.

Верни результат в формате:
1. Порядок выполнения задач.
2. Рекомендуемое время на каждую задачу.
3. Какие задачи нужно сделать сегодня.
4. Какие задачи можно перенести.
5. Краткие рекомендации по организации рабочего процесса.
```

### 6.1. Назначение этого промпта

Этот промпт ориентирован не только на сортировку задач, но и на практическое планирование рабочего дня. Он делает результат полезным для пользователя с точки зрения реальной организации времени.

## 7. Комбинированный промпт для учебного проекта

Для упрощенного варианта проекта можно использовать один объединенный промпт, который выполняет сразу две задачи: анализирует задачи и формирует план.

```text
Ты ИИ-менеджер задач для фрилансера.

Проанализируй список задач пользователя и выполни следующие действия:
1. Определи приоритет каждой задачи.
2. Объясни, почему задача получила такой приоритет.
3. Отсортируй задачи в порядке выполнения.
4. Составь краткий план работы на день.
5. Дай рекомендации по организации нагрузки.

Учитывай:
- дедлайны;
- важность задач;
- наличие ссылки на GitHub-репозиторий проекта;
- примерную длительность;
- зависимости между задачами;
- доступное рабочее время пользователя.

Верни ответ в структурированном виде со следующими разделами:
1. Задачи по приоритету.
   Для каждой задачи укажи текущий статус.
2. План на день.
3. Рекомендации.
```

Такой подход удобен для демонстрационного прототипа, когда важно сократить количество отдельных обращений к API.

## 8. Рекомендации по качеству промптов

При разработке промптов важно соблюдать следующие принципы:

- четко задавать роль модели;
- формулировать одну понятную цель или несколько последовательных целей;
- явно перечислять критерии анализа;
- задавать желаемый формат ответа;
- избегать слишком общих инструкций;
- использовать одинаковую структуру промптов для стабильности результатов.

Дополнительно можно включить:

- примеры входных данных;
- примеры ожидаемого ответа;
- ограничения на стиль и длину ответа;
- инструкцию возвращать данные в JSON.
