# Этап 4. Реализация логики обработки задач

## 1. Цель этапа

Цель данного этапа заключается в реализации основной логики работы ИИ-системы, которая отвечает за обработку списка задач пользователя. На этом этапе система должна уметь анализировать задачи, определять их приоритет и формировать план выполнения.

Именно здесь проект переходит от проектирования и интеграции к прикладной логике, которая создает основную ценность для пользователя. Для фрилансера важно не просто хранить список задач, а получать понятную последовательность действий и рекомендации по организации рабочего времени.

## 2. Общая логика обработки задач

После того как пользователь ввел список задач через интерфейс, система выполняет последовательную обработку данных. Общая схема выглядит следующим образом:

1. Получение списка задач от пользователя.
2. Предварительная проверка и нормализация данных.
3. Анализ содержания задач.
4. Приоритизация задач по заданным критериям.
5. Формирование плана выполнения.
6. Вывод результата пользователю.

Таким образом, каждая следующая стадия опирается на результаты предыдущей, а итогом становится готовый план работы.

## 3. Анализ списка задач

### 3.1. Назначение анализа

Анализ списка задач нужен для того, чтобы система могла понять содержание входных данных и подготовить их к дальнейшей обработке. На этом этапе важно определить характеристики каждой задачи, которые будут влиять на приоритет и порядок выполнения.

### 3.2. Что анализирует система

При анализе система должна учитывать:

- название задачи;
- описание задачи;
- проект или клиента;
- ссылку на GitHub-репозиторий проекта;
- тип задачи;
- дедлайн;
- уровень важности;
- примерную длительность;
- текущий статус;
- зависимости между задачами;
- дополнительные комментарии.

### 3.3. Основные действия на этапе анализа

На этапе анализа система выполняет следующие действия:

1. Проверяет полноту данных по каждой задаче.
2. Определяет, есть ли у задачи дедлайн и насколько он близок.
3. Анализирует важность задачи по данным пользователя.
4. Оценивает длительность выполнения.
5. Выявляет зависимости между задачами.
6. Определяет, какие задачи уже находятся в работе.
7. Группирует задачи по проектам или типам.
8. Учитывает наличие GitHub-ссылки как части проектного контекста задачи.

### 3.4. Результат этапа анализа

На выходе система должна получить структурированное представление списка задач, в котором по каждой задаче определены:

- степень срочности;
- степень важности;
- примерная трудоемкость;
- наличие связей с другими задачами;
- контекст выполнения.

Этот результат используется на следующем этапе для расчета приоритетов.

## 4. Приоритизация задач

### 4.1. Назначение приоритизации

Приоритизация позволяет определить, какие задачи пользователь должен выполнить в первую очередь, а какие можно отложить. Это особенно важно для фрилансеров, которые одновременно ведут несколько проектов и ограничены по времени.

### 4.2. Критерии приоритизации

Система должна учитывать несколько критериев:

- срочность задачи;
- важность задачи;
- влияние задачи на другие задачи;
- влияние на клиента или доход;
- длительность выполнения;
- текущий статус;
- риск просрочки.

### 4.3. Логика определения приоритета

Пример базовой логики приоритизации:

- если дедлайн близкий и задача имеет высокую важность, она получает высокий приоритет;
- если задача блокирует выполнение других задач, ее приоритет повышается;
- если задача уже находится в работе, ее может быть целесообразно завершить раньше;
- если задача короткая, но срочная, ее можно выполнить в начале рабочего дня;
- если задача важная, но срок еще не близок, она может быть отнесена к среднему приоритету;
- если задача не срочная и не влияет на другие процессы, ее можно перенести.

### 4.4. Возможные уровни приоритета

Для удобства интерпретации система может использовать три основных уровня приоритета:

- `high` - задачи, которые необходимо выполнить в первую очередь;
- `medium` - задачи, которые важны, но могут быть выполнены после критичных;
- `low` - задачи, которые можно отложить без существенных последствий.

### 4.5. Дополнительная интерпретация приоритета

Помимо уровня приоритета, система должна формировать краткое объяснение, почему задача получила именно такую оценку. Например:

- высокий приоритет из-за близкого дедлайна и высокой важности;
- средний приоритет, так как задача важная, но срок выполнения позволяет перенести ее на следующий день;
- низкий приоритет, потому что задача не срочная и не влияет на другие процессы.

Такое объяснение повышает прозрачность работы системы и доверие пользователя к рекомендациям.

## 5. Формирование плана выполнения задач

### 5.1. Назначение этапа

После анализа и приоритизации система должна сформировать понятный и реалистичный план выполнения задач. Это означает, что задачи нужно не просто отсортировать, а распределить в логичной последовательности с учетом доступного времени.

### 5.2. Что учитывается при формировании плана

При построении плана система должна учитывать:

- доступное количество рабочих часов;
- порядок приоритетов;
- длительность задач;
- зависимости между задачами;
- текущий статус задач;
- необходимость не перегружать рабочий день.

### 5.3. Логика формирования плана

Общая логика может быть следующей:

1. Сначала выбираются задачи с высоким приоритетом.
2. Из них в начало ставятся самые срочные и короткие задачи либо задачи-блокеры.
3. Затем в план включаются задачи среднего приоритета, если остается рабочее время.
4. Низкоприоритетные задачи переносятся на более поздний период.
5. Если суммарное время превышает доступный рабочий день, система предлагает перенос части задач.

### 5.4. Пример построения дневного плана

Если пользователь указал, что располагает 6 рабочими часами, система может:

- включить в план две срочные задачи на 2 и 3 часа;
- добавить одну короткую задачу на 1 час;
- остальные задачи предложить перенести на следующий день.

Таким образом, план остается реалистичным и выполнимым.

### 5.5. Структура итогового плана

Итоговый план может содержать:

- порядок выполнения задач;
- список задач со статусами;
- ссылки на GitHub-репозитории, связанные с задачами;
- рекомендуемое время на каждую задачу;
- список задач на сегодня;
- список задач на потом;
- рекомендации по организации рабочего процесса.

Пример структуры результата:

| Порядок | Задача | Приоритет | Время | Рекомендация |
|---|---|---|---|---|
| 1 | Исправить баг в форме заявки | high | 2 ч | Выполнить в начале дня |
| 2 | Подготовить текст для лендинга | high | 3 ч | Выполнить после первой задачи |
| 3 | Подготовить отчет по рекламной кампании | medium | 1 ч | Начать сегодня, завершить при наличии времени |

Дополнительно система может выводить отдельный список задач с текущими статусами, например: "к выполнению", "в процессе", "заблокировано", "завершено". Это делает итоговый результат более понятным для пользователя и упрощает контроль рабочего процесса.

## 6. Реализация логики в системе

### 6.1. Роль Python-модуля

Python-модуль в проекте отвечает за координацию логики обработки задач. Он должен:

- принять данные из интерфейса;
- выполнить предварительную валидацию;
- передать данные в LLM;
- получить результат анализа;
- преобразовать ответ в удобную структуру;
- вернуть результат пользователю.

### 6.2. Роль LLM в логике обработки

LLM выполняет интеллектуальную часть обработки:

- интерпретирует текстовые описания задач;
- учитывает контекст и дополнительные комментарии;
- оценивает срочность и важность;
- формирует аргументированную приоритизацию;
- помогает составить последовательность выполнения.

### 6.3. Комбинированный подход

Наиболее рациональным для проекта является комбинированный подход:

- Python выполняет проверку, нормализацию и техническую обработку данных;
- LLM выполняет смысловой анализ и генерацию рекомендаций.

Такой подход повышает надежность системы и делает результаты более предсказуемыми.

## 7. Пример логики обработки одной задачи

Рассмотрим задачу:

- название: "Исправить баг в форме заявки";
- дедлайн: завтра;
- важность: высокая;
- длительность: 2 часа;
- комментарий: "Ошибка влияет на получение заявок от клиента".

Система анализирует эту задачу и может сделать следующий вывод:

- задача срочная;
- задача влияет на бизнес-результат;
- задача короткая по времени;
- задачу целесообразно выполнить в первую очередь.

Следовательно, система присваивает ей высокий приоритет и включает в начало рабочего плана.

## 8. Ожидаемый результат этапа

После выполнения этапа должны быть реализованы:

- логика анализа списка задач;
- механизм определения приоритета;
- правила формирования последовательности выполнения;
- возможность вывода списка задач со статусами;
- возможность хранения и вывода GitHub-ссылки проекта в задаче;
- модель выдачи итогового плана;
- связка между Python-модулем и LLM для обработки задач.

## 9. Вывод

На этапе реализации логики обработки задач была описана основная функциональная часть системы, которая отвечает за анализ задач, их приоритизацию и формирование плана выполнения. Именно этот этап превращает входные данные пользователя в практический результат: понятный список приоритетов и последовательность действий. За счет сочетания технической обработки данных в Python и интеллектуального анализа с помощью LLM система может выдавать полезные рекомендации для организации работы фрилансера.
