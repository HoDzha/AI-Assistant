# Этап 1. Проектирование системы

## 1. Цель этапа

На первом этапе проектирования необходимо определить, в каком формате пользователь будет передавать задачи в систему, а также описать сценарий анализа этих задач. Этот этап является базовым, так как от структуры входных данных зависит качество анализа, приоритизации и формирования плана работы.

В рамках проекта разрабатывается ИИ-менеджер задач для фрилансеров. Система должна принимать список задач, учитывать контекст по каждой задаче и передавать эти данные в LLM для последующей обработки.

## 2. Формат входных данных

### 2.1. Требования к входным данным

Чтобы система могла корректно анализировать задачи, входные данные должны быть:

- понятными для пользователя;
- удобными для обработки в Python;
- пригодными для передачи в LLM через API;
- достаточно структурированными для дальнейшей приоритизации и планирования.

Оптимальным форматом входных данных для проекта является список задач в формате JSON. Такой формат легко валидировать, хранить и передавать между компонентами системы.

### 2.2. Структура входных данных

Каждая задача представляется как отдельный объект со следующими полями:

- `id` - уникальный идентификатор задачи;
- `title` - краткое название задачи;
- `description` - подробное описание задачи;
- `project` - проект или клиент, к которому относится задача;
- `github_url` - ссылка на GitHub-репозиторий проекта;
- `type` - тип задачи;
- `deadline` - срок выполнения;
- `priority_hint` - важность задачи по мнению пользователя;
- `estimated_hours` - примерная оценка времени выполнения;
- `status` - текущее состояние задачи;
- `dependencies` - список задач, от которых зависит выполнение;
- `notes` - дополнительные комментарии пользователя.

### 2.3. Описание полей

| Поле | Тип данных | Описание |
|---|---|---|
| `id` | string | Уникальный идентификатор задачи |
| `title` | string | Краткое название задачи |
| `description` | string | Подробное описание содержания задачи |
| `project` | string | Название проекта или клиента |
| `github_url` | string | Ссылка на GitHub-репозиторий, связанный с задачей |
| `type` | string | Категория задачи, например: дизайн, текст, разработка, коммуникация |
| `deadline` | string | Дата и время дедлайна в формате `YYYY-MM-DD HH:MM` |
| `priority_hint` | string | Уровень важности от пользователя: `low`, `medium`, `high` |
| `estimated_hours` | number | Оценка трудозатрат в часах |
| `status` | string | Статус задачи: `new`, `in_progress`, `done` |
| `dependencies` | array | Идентификаторы связанных задач |
| `notes` | string | Дополнительные комментарии |

### 2.5. Почему выбран именно такой формат

Выбранный формат удобен по нескольким причинам:

- он подходит для ввода через форму, Telegram-бота, веб-интерфейс или API;
- его легко обрабатывать с помощью Python;
- структура данных прозрачна для LLM;
- можно масштабировать систему, добавляя новые поля без изменения общей логики;
- можно хранить вместе с задачей ссылку на GitHub-репозиторий проекта;
- формат позволяет учитывать не только сами задачи, но и рабочий контекст пользователя.

## 3. Проектирование сценария анализа задач

### 3.1. Цель сценария анализа

Сценарий анализа задач нужен для того, чтобы преобразовать исходный список задач в структурированный и понятный план действий. На этом этапе система должна не просто принять список задач, а определить, какие из них более важны, какие нужно выполнить раньше и как лучше распределить работу.

### 3.2. Общая логика сценария

Сценарий анализа задач включает несколько последовательных этапов:

1. Пользователь вводит список задач и дополнительную информацию.
2. Система проверяет корректность структуры входных данных.
3. Задачи проходят предварительную обработку.
4. Подготовленные данные передаются в LLM через API.
5. LLM анализирует задачи по заданному промпту.
6. Система получает структурированный ответ.
7. На основе ответа формируется итоговый список приоритетов и план работы.

### 3.3. Подробное описание сценария

#### Шаг 1. Получение данных от пользователя

Пользователь передает:

- список задач;
- сроки выполнения;
- важность задач;
- ссылку на GitHub-репозиторий проекта при необходимости;
- примерную длительность;
- при необходимости дополнительный контекст.

Источником данных может быть:

- веб-форма;
- интерфейс приложения;
- чат-бот;
- автоматизированный сценарий в `n8n`.

#### Шаг 2. Валидация входных данных

На этом этапе система проверяет:

- заполнено ли название задачи;
- указан ли дедлайн;
- корректно ли задан формат даты;
- есть ли дублирующиеся идентификаторы;
- заполнены ли обязательные поля.

Если данные введены некорректно, система должна вернуть понятное сообщение об ошибке и предложить исправить ввод.

#### Шаг 3. Предварительная обработка

Перед отправкой в LLM система выполняет базовую нормализацию:

- удаляет пустые задачи;
- приводит даты к единому формату;
- преобразует текстовые оценки важности в стандартные значения;
- группирует задачи по проектам;
- определяет просроченные или срочные задачи;
- отмечает задачи с зависимостями.

Этот шаг снижает вероятность ошибок и улучшает качество ответа модели.

#### Шаг 4. Анализ задач с помощью LLM

После подготовки данных система отправляет их в LLM вместе с промптом, в котором задается роль ассистента по управлению задачами.

Модель должна:

- проанализировать список задач;
- определить срочность и важность;
- выявить логические связи между задачами;
- предложить порядок выполнения;
- сформировать краткие рекомендации по организации работы.

#### Шаг 5. Получение структурированного результата

Результат анализа должен возвращаться в структурированном виде. Например, по каждой задаче система может получить:

- итоговый приоритет;
- причину назначения приоритета;
- рекомендуемый порядок выполнения;
- рекомендации по времени выполнения;
- комментарий по рискам, если дедлайн слишком близкий.

#### Шаг 6. Формирование итогового плана

После получения ответа от LLM система формирует итоговый результат для пользователя:

- список задач в порядке приоритета;
- список задач с отображением текущих статусов;
- краткий план работы на день или неделю;
- рекомендации по началу работы с наиболее срочных и важных задач;
- предложения по распределению нагрузки.

## 4. Логика принятия решений системой

Для анализа задач система может учитывать следующие критерии:

- срочность задачи;
- важность задачи;
- длительность выполнения;
- наличие зависимостей;
- принадлежность к активному проекту;
- влияние задачи на клиента или доход.

На основе этих критериев LLM определяет относительный приоритет каждой задачи.

Пример базовой логики:

- если дедлайн близкий и задача важная, приоритет высокий;
- если задача зависит от другой задачи, сначала должна быть выполнена зависимость;
- если задача короткая и срочная, ее можно выполнить в начале рабочего дня;
- если задача важная, но несрочная, ее можно запланировать в отдельный блок времени.

## 5. Ожидаемый результат этапа

Результатом этапа проектирования являются:

- определенный формат входных данных;
- описание обязательных и дополнительных полей задачи;
- пример структуры данных в JSON;
- спроектированный сценарий анализа задач;
- зафиксированная возможность вывода списка задач со статусами;
- зафиксированная возможность хранения ссылки на GitHub-репозиторий в задаче;
- понимание того, как данные будут передаваться в LLM и как будет формироваться итоговый результат.

## 6. Вывод

На первом этапе был определен формат входных данных для ИИ-менеджера задач фрилансера и спроектирован сценарий анализа задач. В качестве основного формата выбран JSON, так как он удобен для программной обработки и интеграции с LLM через API. Также был описан пошаговый сценарий, по которому система получает, проверяет, анализирует и преобразует задачи пользователя в структурированный план работы. Этот этап создает основу для следующих шагов проекта: разработки промптов, интеграции модели и реализации рабочего прототипа системы.
