# Этап 2. Разработка интерфейса взаимодействия

## 1. Цель этапа

На данном этапе необходимо спроектировать и описать интерфейс взаимодействия пользователя с системой. Основная задача этапа заключается в том, чтобы обеспечить удобный ввод задач и корректную передачу этих данных в ИИ-систему для последующего анализа.

Для проекта "ИИ-менеджер задач для фрилансеров" интерфейс является важной частью, так как именно через него пользователь передает список задач, сроки, важность и дополнительный контекст. Если интерфейс будет неудобным или перегруженным, качество взаимодействия с системой снизится.

## 2. Требования к интерфейсу

Интерфейс должен соответствовать следующим требованиям:

- быть простым и понятным для пользователя;
- обеспечивать быстрый ввод задач;
- поддерживать ввод нескольких задач;
- позволять указывать ключевые параметры задачи;
- передавать данные в структурированном виде;
- обеспечивать базовую проверку корректности данных до отправки в систему.

С учетом целей проекта оптимальным решением является веб-интерфейс с формой ввода задач. Такой вариант проще всего реализовать с помощью Python и при необходимости масштабировать в полноценное веб-приложение.

## 3. Реализация интерфейса для ввода задач

### 3.1. Общая концепция интерфейса

Пользовательский интерфейс должен содержать форму, в которой пользователь может:

- добавить одну или несколько задач;
- указать название и описание задачи;
- выбрать проект или клиента;
- задать срок выполнения;
- отметить важность задачи;
- указать примерную длительность;
- при необходимости оставить комментарий.

Интерфейс может быть реализован в одном из следующих вариантов:

- веб-страница с HTML-формой;
- статический интерфейс на `HTML/CSS/JavaScript`;
- чат-интерфейс;
- форма, связанная с автоматизацией в `n8n`.

Для учебного проекта наиболее рационально выбрать веб-форму или статический интерфейс на `HTML/CSS/JavaScript`, так как они позволяют быстро создать рабочий прототип без сложной frontend-разработки.

### 3.2. Состав формы ввода

Для каждой задачи в интерфейсе рекомендуется использовать следующие поля:

| Поле | Назначение |
|---|---|
| Название задачи | Краткое описание задачи |
| Подробное описание | Уточнение содержания задачи |
| Проект / клиент | Привязка задачи к конкретному проекту |
| Тип задачи | Категория: разработка, текст, дизайн, аналитика и т.д. |
| Дедлайн | Срок выполнения задачи |
| Важность | Оценка приоритета со стороны пользователя |
| Оценка времени | Примерная длительность выполнения |
| Статус | Новая, в работе, завершена |
| Комментарий | Дополнительная информация |

### 3.3. Логика работы формы

Работа формы может быть организована следующим образом:

1. Пользователь открывает интерфейс.
2. Заполняет поля по одной задаче.
3. При необходимости добавляет еще задачи.
4. Нажимает кнопку отправки данных.
5. Система проверяет корректность заполнения.
6. После успешной проверки данные передаются в модуль обработки.

### 3.4. Требования к удобству использования

Чтобы интерфейс был удобен для фрилансера, важно предусмотреть:

- минимальное количество обязательных полей;
- понятные подписи полей;
- единый формат даты и времени;
- кнопки добавления и удаления задачи;
- отображение ошибок при некорректном вводе;
- наглядное подтверждение успешной отправки данных.

Это особенно важно, так как пользователь может ежедневно работать с большим количеством задач, и интерфейс не должен замедлять его работу.

## 4. Настройка передачи данных системе

### 4.1. Общая логика передачи данных

После ввода задач интерфейс должен передавать информацию в систему обработки. Передача данных может происходить по следующей схеме:

Интерфейс пользователя  
↓  
Модуль обработки на Python  
↓  
Подготовка и валидация данных  
↓  
Передача запроса в LLM через API  
↓  
Получение результата  
↓  
Отображение ответа пользователю

Таким образом, интерфейс не выполняет анализ самостоятельно, а служит средством сбора и передачи данных в основную систему.

### 4.2. Формат передачи данных

После заполнения формы данные должны быть преобразованы в JSON-структуру. Это позволяет:

- унифицировать формат обмена;
- упростить обработку в Python;
- передавать данные в API без дополнительного преобразования;
- использовать один и тот же формат на всех этапах системы.

Пример структуры передаваемых данных:

```json
{
  "freelancer_name": "Иван",
  "workday_hours": 6,
  "tasks": [
    {
      "id": "T1",
      "title": "Подготовить текст для лендинга",
      "description": "Написать текст для первого экрана и блока преимуществ",
      "project": "Клиент Alpha",
      "type": "copywriting",
      "deadline": "2026-03-26 18:00",
      "priority_hint": "high",
      "estimated_hours": 3,
      "status": "new",
      "dependencies": [],
      "notes": "Сначала уточнить требования клиента"
    }
  ]
}
```

### 4.3. Валидация данных перед отправкой

Перед передачей данных системе необходимо выполнить базовую проверку:

- заполнено ли название задачи;
- указан ли дедлайн;
- выбрано ли значение важности;
- корректно ли введено время выполнения;
- нет ли пустых задач в списке.

Если данные заполнены неверно, интерфейс должен сообщить об ошибке до отправки запроса. Это снижает нагрузку на систему и уменьшает вероятность получения некорректного результата от модели.

### 4.4. Передача данных в Python-модуль

После успешной проверки данные могут передаваться в Python-модуль, который:

- принимает структуру JSON;
- выполняет дополнительную серверную валидацию;
- формирует промпт для LLM;
- отправляет запрос к модели через API;
- получает ответ и возвращает его в интерфейс.

В зависимости от реализации это может быть:

- обработчик формы в `Flask`;
- endpoint в `FastAPI`;
- клиентская логика и отправка запросов из статического `HTML/CSS/JavaScript` интерфейса;
- автоматизированный сценарий в `n8n`.

### 4.5. Передача данных в LLM через API

На уровне интеграции с ИИ-моделью система должна:

- собрать данные пользователя;
- встроить их в подготовленный промпт;
- отправить запрос через API;
- получить структурированный ответ;
- отобразить пользователю итоговый результат.

В ответе система может получить:

- отсортированный список задач;
- уровень приоритета для каждой задачи;
- краткое объяснение приоритета;
- план работы на день или неделю;
- рекомендации по организации нагрузки.

## 5. Пример пользовательского сценария

Ниже приведен пример типового сценария взаимодействия:

1. Фрилансер открывает веб-интерфейс.
2. Указывает свое имя и доступное количество рабочих часов.
3. Добавляет несколько задач от разных клиентов.
4. Для каждой задачи указывает дедлайн, важность и оценку времени.
5. Нажимает кнопку "Проанализировать задачи".
6. Интерфейс преобразует данные в JSON.
7. Данные передаются в Python-модуль.
8. Модуль отправляет запрос в LLM.
9. Система получает ответ и выводит:
   список задач по приоритету, рекомендации и план выполнения.

## 6. Техническая реализация этапа

Для учебного проекта можно использовать следующую схему:

- `Python` как основной язык разработки;
- `Streamlit` или `Flask` для интерфейса;
- `requests` или библиотеку SDK для обращения к API модели;
- JSON как внутренний формат передачи данных;
- при необходимости `n8n` для автоматизации сценариев.

Такой подход позволяет быстро создать понятный прототип и продемонстрировать практические навыки интеграции интерфейса с AI-моделью.

## 7. Ожидаемый результат этапа

После выполнения этапа должны быть определены:

- формат пользовательского интерфейса;
- набор полей для ввода задач;
- логика добавления и отправки задач;
- схема передачи данных в систему;
- подход к валидации данных;
- способ интеграции интерфейса с Python-модулем и LLM API.

## 8. Вывод

На этапе разработки интерфейса взаимодействия была определена форма ввода задач и логика передачи данных в систему. В качестве базового решения реализован статический веб-интерфейс на `HTML/CSS/JavaScript`, позволяющий пользователю быстро вводить задачи, хранить GitHub-ссылки и отправлять их на анализ в `FastAPI` backend. Также был описан механизм преобразования введенных данных в JSON-формат, их валидации и дальнейшей передачи в Python-модуль, который взаимодействует с LLM через API. Этот этап формирует связующее звено между пользователем и интеллектуальной логикой системы.
